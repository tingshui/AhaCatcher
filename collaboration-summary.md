# 如何更好地与 AI（Cursor）协作 —— 沟通与习惯

> 本文**以「人如何与 AI 配合」为主**；技术实现细节见 `README.md`、`how-to-use.md`。你可按需增删本页。

---

## 1. 先说清「要什么」和「不要什么」

- **目标**：一句话说清结果（例如「点一个按钮看后台同步状态」「通知里只要三样信息」）。越具体，越不需要来回猜。
- **边界**：若有限制（只做 Mac、必须用 ntfy、不能上 App Store 等）尽早说，避免方案跑偏。
- **成功标准**：你希望「怎样算做完」——能演示的路径、能接受的最简版本，会显著减少返工。

---

## 2. 给足上下文，但不必一次写成长文

- **贴终端/报错**：出错时把**完整命令 + 关键错误行**贴出来（或 `@` 引用终端文件），比只描述「报错了」有效得多。
- **指到文件**：用 `@文件` 或路径，让 AI 直接读代码/配置，减少口述偏差。
- **会话连续性**：同一条需求里若你**改主意或加一句**，尽量说明是「在上一句基础上改」还是「新任务」，AI 会少误解。

---

## 3. 迭代方式：先跑通，再打磨

- **小步**：先实现能用的路径（例如先手动跑脚本，再 launchd），再优化体验（状态页、文案）。
- **显式纠正**：若结果不对，直接说「保留 A，改掉 B」或「和上次比只要多 X」，比笼统「再改改」省轮次。
- **接受「需要你在本机点一下」**：例如系统**完全磁盘访问**、launchd 权限，AI 可写 plist 和说明，但**勾选设置**必须你操作——提前预期这种分工，不会觉得「AI 没做完」。

---

## 4. 利用项目里的「长期记忆」

- **`.cursorrules`**：你们约定了语言、Scratchpad、工具路径（`tools/shared`、`ahacatcher/scripts`）等。重要约定写进规则，**下一轮对话仍会遵守**。
- **Lessons**：把「踩过一次就不想再踩的坑」写成一条 Lesson，以后同类问题会更快对齐。
- **本地文档**：`how-to-use.md`、`collaboration-summary.md` 这类文件是给你自己看的「备忘」，不必和 AI 重复全文，需要时 `@` 一下即可。

---

## 5. 沟通语言与风格

- 你**默认用中文**时，面向你的说明会用中文；**命令、文件名、API** 保持英文是习惯，不必混翻。
- 若你希望**整份设计文档/注释用英文**或**双语**，可以直接说，便于以后给其他人看。

---

## 6. 什么时候拆成多个请求

- **范围大**（重构目录 + 改文档 + 改 launchd）时，可以拆成：「先只做目录移动与引用更新」，再「再补 README」。
- **探索性**（「这样设计好不好」）与**落地性**（「请直接改代码」）可以分开问，前者适合讨论，后者适合 Agent 模式连续改文件。

---

## 7. 需求与项目：什么一开始就该做好（反思）

> 以下是对本类项目**不留情面的复盘**：针对**做法与预期**，便于下一个项目前置对齐。不是否定个人，而是收紧流程。

### 7.1 需求一开始就该钉死的三件事

- **「通知成功」≠「数据落在你以为的地方」**
  若某条链路（例如 quick-memo）最初只承诺「转写 + 分类 + 推送」，则**不会**自动出现在 Ideas 里——除非一开始就把 **「最终落盘路径、格式、是否必须持久化」** 写进需求。
  **教训：** 对每条用户路径写清：**输入 → 处理 → 落在哪里、什么格式**。

- **多入口要画清**
  「网页录音保存」与「语音备忘录 → Mac → 扫描/快捷指令」是不同入口；若早期就画：**入口 A / 入口 B → 是否共用 Ideas、是否共用 ntfy**，可减少「我以为和网页一样」的落差。
  **教训：** 多入口产品，先画**数据流**（哪怕手绘），再写功能。

### 7.2 环境与权限：应更早系统性处理

- **完全磁盘访问、launchd、venv Python 真实路径** 等属于 **Apple/系统规则**，不是「代码写错就能在 IDE 里解决」。若一开始就有 **「后台任务用哪个二进制、FDA 要加谁」** 的清单，会少很多轮试错。
  **教训：** 涉及 TCC/沙盒/后台时，先做 **权限检查清单**，再写自动化。

### 7.3 配置与仓库卫生

- **`.env`、密钥、路径** 应集中在 **单一真源**（`README` 表格 + `.env.example`），每增一个变量就写清 **默认值与「不设会怎样」**。
  **教训：** 避免「只记在聊天里」导致漏配、错配。

### 7.4 验证习惯

- **以「磁盘/API 返回」为准**，不要**只以 ntfy 为准**。推送只说明「某段逻辑跑过」；文件是否在 Ideas、API 是否返回 `saved_to_ideas` 是另一回事。
  **教训：** 为关键路径定 **10 秒自检**（例如列目录、`curl` 看字段）。

### 7.5 理想情况下「从一开始就该做好的」一览

| 维度 | 理想做法 |
|------|----------|
| 需求 | 每条路径：**输入 → 处理 → 落盘位置 → 通知** 写清 |
| 架构 | 多入口是否 **共用同一套保存与 YAML**，尽早定 |
| 运维 | launchd / FDA / 定时间隔 **文档化**，并与 `.env` 一致 |
| 验证 | **成功定义**含「持久化结果可见」，不仅「推送收到」 |
| 沟通 | 大需求 **分阶段**：先跑通再 polish；**需求澄清尽量前置** |

### 7.6 一句实话

很多摩擦来自 **Voice Memos + iCloud + 本机脚本 + 第三方 API** 本身就很碎，未必是「粗心」。最值得补的一课是：**先把「数据最终放哪」说死，再选技术方案。**
